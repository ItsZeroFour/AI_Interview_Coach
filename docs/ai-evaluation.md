# Карточка оценки AI-компонента

**Название проекта:** InterviewAI
**AI-компонент:** Адаптивный агент-интервьюер
**Тип адаптации:** AI-агент + prompt engineering (routing, few-shot, self-reflection, chain-of-thought)
**Baseline:** Единичный вызов LLM с промптом «проведи собеседование» без контекста резюме/вакансии
**Improved:** Агентный пайплайн с анализом резюме, routing по типам вопросов, few-shot примерами, self-reflection и CoT-оценкой

## 1. Метрики

**Основные:**
- Win-rate (предпочтение improved vs baseline в слепом сравнении)
- Relevance score (релевантность вопросов к вакансии, 1-5)
- Format compliance (% ответов в заданном JSON-формате)
- Latency (время генерации одного вопроса)

**Дополнительные:**
- Адаптивность: вопросы меняются в зависимости от ответов
- Покрытие типов вопросов: behavioral, technical, situational
- Полезность фидбека по экспертной оценке: 1-5

## 2. Eval-set

- Количество примеров: 20 тест-кейсов
- Источник: вручную собранные пары (резюме + вакансия + ответы разного качества)
- Сценарии: junior/middle кандидат, Frontend/Backend/QA вакансии, сильные/слабые/нерелевантные ответы
- Пограничные случаи: пустое резюме, нерелевантная вакансия, ответ не по теме

## 3. Сравнение

| Метрика | Baseline | Improved | Изменение |
|---------|----------|----------|-----------|
| Win-rate | - | 78% | Improved выигрывает в 78% слепых сравнений |
| Relevance | 2.3/5 | 4.1/5 | +78% - вопросы учитывают резюме и вакансию |
| Format compliance | 45% | 92% | +47pp - JSON-формат стабильно соблюдается |
| Latency | ~2s | ~3s | +1s - допустимо для интерактива |
| Адаптивность | Нет | Да | Baseline повторяет типы, improved чередует |

## 4. Качественные примеры

**Пример 1: Baseline задаёт generic вопрос, improved - контекстный**
- Baseline: «Расскажите о себе»
- Improved: «В вашем резюме указан опыт с React и Redux. На вакансии требуется знание микрофронтендов. Расскажите, как бы вы организовали переход к микрофронтендной архитектуре в существующем React-проекте?»

**Пример 2: Improved адаптирует сложность**
- После слабого ответа кандидата (score 2/5) improved переключается с technical на behavioral, задавая более простой вопрос об опыте
- Baseline продолжает задавать сложные технические вопросы

**Пример 3: Improved даёт структурированный фидбек**
- Baseline: пустая оценка или «хорошо»
- Improved: JSON с баллом 3/5, strengths: [«Упомянул конкретный проект»], improvements: [«Добавьте измеримые результаты - на сколько % улучшилась производительность»], comment: «Ответ показывает опыт, но не хватает цифр и конкретных метрик успеха»

## 5. Вывод

**Что реально улучшилось:**
- Релевантность вопросов выросла с 2.3 до 4.1 за счёт контекста резюме + вакансии
- Стабильность JSON-формата выросла с 45% до 92% благодаря раздельным системным промптам и json_object mode
- Self-reflection убирает шаблонные вопросы вроде «расскажите о себе»

**За счёт чего:**
- Few-shot примеры хороших и плохих вопросов дают модели чёткие ориентиры
- Раздельные роли (интервьюер vs эксперт) улучшают качество оценки
- Chain-of-thought в evaluation заставляет модель анализировать перед оценкой

**Что остаётся проблемой:**
- На слабых моделях (< 7B) качество JSON-парсинга падает
- Latency увеличивается из-за self-reflection (двойной вызов)
- Модель иногда генерирует вопросы на английском при русскоязычном промпте
